# AGENTS.md

## Learned User Preferences

- Never add `Co-authored-by: Cursor` (or any Cursor) trailers to commit messages; the user wants commit messages fully clean.
- Do not provide a list of changes in responses.
- Never create fake/sample data or data generators; treat all data as real or captured by real systems.
- Never ask the user to restart development servers.
- For UI work, keep coloring clean, simple, and non-invasive; lean on the brand slate + single yellow accent rather than heavy/decorative color.
- Keep the codebase unified: do not leave unused, duplicated, or over-complex code from multi-agent history; consolidate and remove dead code rather than piling on new files.
- Prefer reframing/inventorying what already exists over adding new features; when asked to "reimagine," audit current capabilities and decide keep/reframe/change rather than building more.
- Keep PaveRate branding visible even when an org sets a custom logo (orgs can customize name/logo/default values, but PaveRate identity stays present).
- Don't state deployment/infrastructure facts (e.g. which branch maps to the `dev.paverate.com` subdomain, what a deploy is serving) as fact from assumption or CLI metadata — verify directly (probe the live endpoint/response) before asserting; the user pushes back hard on guessed claims presented as truth.

## Learned Workspace Facts

- The shell is PowerShell on Windows: bash idioms like `head`, `ls -la`, `du`, and inline `$VAR=...` env-prefixes fail. Use `Select-Object`, `Get-ChildItem`, and run multi-statement logic from a script file (passing complex `$`-laden commands inline often gets the `$` stripped).
- Build script must stay `wrangler types && vite build` (no `--check`); `--check` breaks the Cloudflare Pages build because `worker-configuration.d.ts` does not exist pre-build.
- All values, spec rates, constants, theme tokens, and labels live in the single config `src/lib/config/paverate.yaml`; formulas live in `src/lib/config/formulas.ts`. No magic numbers in code — every value is read by its Validation Matrix ID (e.g. `CONST.LB_PER_TON`, `TACK.NEW_AC`).
- Calculations must be traceable to GDOT specs via `docs/validation-matrix.md` and `docs/VALIDATION.md`; verified spec excerpts are saved under `docs/spec-excerpts/`.
- GDOT spec PDFs live in `docs/` (2021 Standard Specs, 2024 Supplemental, `cm001` Construction Manual); `tools/spec_search.py` (uses `pypdf`, in `requirements.txt`) searches/dumps them. The PDF page index is offset from the printed footer page number (e.g. printed p.449 = PDF p.459).
- Brand identity: dark slate `#2e3b46` + single yellow accent `#f2c037`; dark-mode default, high contrast for outdoor use, mobile-first with 48px+ touch targets. Brand/PWA assets are generated by `tools/make_brand_assets.py` (Pillow/`PIL`) from source transparent PNGs in `branding/`, emitting trimmed icons/logos to `static/icons/` and `static/`; in-app logos are inline brand-colored SVG (slate badge + yellow road/lane) to keep transparent corners sharp.
- Stack: SvelteKit 5 (runes) + `@sveltejs/adapter-cloudflare`, PWA via `@vite-pwa/sveltekit`, YAML imported at build time via `@rollup/plugin-yaml`; deploys to Cloudflare Pages from `main`. Accounts/orgs/jobsites are a planned optional cloud layer (Workers + D1 + R2), never gating offline calculator use. Runes gotcha: never name a component prop `state` — it shadows the `$state` rune so `$state(...)` is misread as a store subscription to the prop; rename the prop (e.g. `data`).
- Auth/accounts use a Cloudflare D1 binding (`event.platform.env.DB`) and an R2 binding `ASSETS_BUCKET` (bucket `paverate-org-assets`, used for org logo uploads); both are available under `vite dev` via adapter-cloudflare's platform proxy reading `wrangler.jsonc`. A dev-only login endpoint/button seeds a known dev user+org and is gated behind SvelteKit's `dev` flag (absent in production). `setSessionCookie` (`src/lib/server/auth.ts`) sets `secure: !dev` so local HTTP sessions persist — a hard-coded `Secure` cookie is silently dropped over `http://localhost` and 401s every protected route.
- Local D1 is bootstrapped via npm scripts (no manual per-file applies): `npm run db:local` (apply all `migrations/0*.sql` in sorted order via `scripts/db-local.ps1`), `db:local:reset` (wipe + rebuild), and `db:pull-dev` (export the remote dev D1 to `.wrangler/dev-snapshot.sql` and load it into local, for real-data testing). No D1 mock/refactor — `DbHelper` issues real SQL via `prepare().bind().first/all/run()` and adapter-cloudflare gives `vite dev` a real local SQLite.
- Protected `+server.ts` endpoints must `return error` (not `throw error`) when catching a `Response` thrown by `requireAuth`/`requireOrgRole`/`requireGlobalAdmin`; SvelteKit only special-cases thrown `Redirect`/`HttpError`, so a thrown raw `Response` becomes a fatal 500 instead of the intended 401/403 JSON.
- The product has pivoted from a simple calculator into a full field-paving project-management tool; "Job Sites" are really Projects (config, costs/contract value, crews, equipment, route/geometry, daily logs, loads), with calculators as one piece and most values auto-derived. Keep a standalone one-off calculator too. The DB/APIs already support the PM model; the UI lags it. Orgs can customize name/logo/default values, with logo uploads via the `ASSETS_BUCKET` R2 binding noted above. The `/app` route is "Quick Calc" — it should be pure calculators/inputs only (no daily-log/"Today"/project-management features, which belong to the separate Job Sites/dashboard routes).
- Shared calculator inputs flow through `src/lib/stores/calcContext.svelte.ts`, the single source of truth that merges manual overrides → job-site config → the `job` store (`src/lib/stores/job.svelte.ts`) with source tracking. A value set once (e.g. `lift_thickness`, `road_width`, `course_type`) should be read by EVERY calculator via `calcContext`, not directly from `job` — several cards historically read `job.*` directly and so ignored overrides, which is the bug to standardize away. Genuinely job-only fields (machine, firstPass, waste, truck load) still read from `job`.
- Server-side PDF text extraction uses `pdfjs-serverless` (`getDocument` → `getTextContent`, with `disableFontFace`/`isEvalSupported:false`/`useSystemFonts`), NOT `unpdf`: `unpdf`'s bundled PDF.js works in `wrangler dev` locally but throws `Object.defineProperty called on non-object` ("Serverless PDF.js bundle could not be resolved") in the actual deployed Cloudflare Pages runtime, so it was replaced. Uploaded source PDFs are stored in the `ASSETS_BUCKET` R2 binding (e.g. under `imports/<orgId>/<uuid>.pdf`). The GDOT job-setup / contract-summary parser (`src/lib/server/pdf/parse-gdot.ts`) is 100% deterministic regex/zone parsing (no LLM, leaves unknowns undefined, never invents data): `detectDocumentType()` classifies via sentinel-phrase regex, parsed values are wrapped in a `ParsedField<T>` confidence envelope (high/medium/low) with low-confidence regions collected as `lowConfidenceZones`, and the no-text path returns 422 ("may be a scanned image"). Per `docs/pdf-parsing-design.md` Phase 2, any LLM (e.g. Workers AI) is ONLY ever a fallback over `lowConfidenceZones`/scanned docs overriding `low` fields only — never a replacement for deterministic extraction (preserves spec-traceability). Rendering PDF pages to images (schematics) can NOT run in the Workers runtime (no canvas) — do it client-side with `pdfjs` in the browser and upload the PNGs to R2.
- Migrations live in `migrations/` (NOT the nested `migrations/migrations/`) and are applied by `scripts/db-local.ps1`, which now runs `npx wrangler d1 migrations apply paverate-db --local` (tracked in `d1_migrations`); `-Reset` drops all user objects by name first (the SQLite file is locked while `vite dev` runs). This requires strict sequential numbering with NO duplicates — verify the current head before adding a new file (recent head was `0047`, new files `0048`/`0049`). SQLite has no `ADD COLUMN IF NOT EXISTS`, so keep a bare `ALTER TABLE ... ADD COLUMN` in its own migration file. (Historical multi-agent dupes like `0024`/`0025`/`0046` predate this and should be cleaned up rather than added to.)
- Wrangler is currently authenticated (as `dylan@dlewis.me`, with `d1`/`pages` write scope), so live Cloudflare access (remote D1 migrations, Pages deployment status via the API) works from the shell — the older "expired OAuth token" note is stale. There is exactly ONE D1 database in the account (`paverate-db`, single `DB` binding in `wrangler.jsonc`, no per-environment D1), so `dev.paverate.com` (a Pages preview/branch deploy) and production SHARE the same database — there is no separate "dev D1"; any migration lag is on that one shared DB. Migrations auto-apply only on push to `main` via `.github/workflows/deploy.yml` (now using the tracked `npx wrangler d1 migrations apply paverate-db --remote`, which only runs untracked files and fails loudly — the old "execute every .sql file, swallow errors" loop hid broken migrations); `npm run db:migrate:remote` is the manual equivalent. Check pending with `wrangler d1 migrations list paverate-db --remote`. Watch for 500s on org endpoints (`/api/org/crew-status`, `/api/org/settings`) sharing `getOrgByUserId` → `getUserRole` → `getOrgSettings`.
- PWA `navigateFallback` must stay `null` in `vite.config.ts`: this is an SSR app on Cloudflare with nothing prerendered at `/`, so the default `navigateFallback: '/'` makes Workbox bind a `NavigationRoute(createHandlerBoundToURL('/'))` to a non-precached URL and throw `non-precached-url :: [{"url":"/"}]` on every navigation (non-fatal but noisy). Asset precaching + font/`/api` runtime caching are unaffected. Separately, Leaflet must NOT be listed in `build.rollupOptions.external` in `vite.config.ts` — externalizing it emits a bare `import "leaflet"` the browser can't resolve ("Failed to resolve module specifier 'leaflet'") in the production/preview build (dev hides it because Vite resolves bare specifiers on the fly); keep `ssr.noExternal: ['leaflet']` so it's bundled for both client and SSR.
- GDOT-import projects are multi-scope/multi-mix: each mix carries TWO distinct quantities — `bid_quantity` = "allotted" (contract/Paving-Sections quantity the state/customer pays for) vs `takeoff_tonnage` = "target" (the company's Production-Goals number we aim to place) — plus a `contract_unit_price` (from the matching asphalt bid item) so margin = unit price − our cost/ton. Don't color the target-vs-allotted variance good/bad (placing fewer tons than allotted is just less line revenue, not "more money"; margin comes from cost < unit price). The parser must map mix names to the mix_type enum (not store the name as the type) and detect/report which document was uploaded (Contract Summary vs Job Setup), warning when one is missing.
- App shell layout (`src/lib/components/shell/AppShell.svelte` + `NavSidebar.svelte`) is a responsive CSS grid with breakpoints at 900px (icon-rail sidebar) and 1100px (labeled sidebar); mobile uses a hamburger drawer. The grid MUST use explicit `grid-template-areas` (`nav` spanning all rows, with `weather`/`main`/`footer`/`context` stacked in the content column) — relying on auto-flow scatters main/footer into the sidebar column. The `.sidebar` must be `position: sticky; align-self: start` across the entire ≥900px range (not just 900–1099px), or at ≥1100px it falls back to base `position: fixed`, drops out of grid flow, and leaves an empty reserved column.

## Map Conventions

- The map layer is fully on MapLibre GL JS: the ONLY map layer is `src/lib/components/map-v2/` (lazy, SSR-safe `MapView` + `MapMarker`/`MapPolyline`/`MapPolygon`/`MapGeoJSON`/`MapPopup`/`RoadProgressLayer`). The old `src/lib/components/map/` Leaflet directory has been removed, and `leaflet`/`@types/leaflet` are no longer dependencies (do NOT re-add them or `ssr.noExternal: ['leaflet']`). Build new map features by composing `map-v2` components inside a `<MapView>`; never use raw `L.map()` or any Leaflet API.
- Store geometry as GeoJSON TEXT in D1 (column name: `geometry_geojson`); use `geoJsonToD1()` / `d1ToGeoJson()` from `src/lib/services/mapUtils.ts`.
- Coordinates: `map-v2` component props take `[lat, lng]` (the component flips internally); geometry is stored as GeoJSON `[lng, lat]` per RFC 7946, and `RoadProgressLayer` consumes stored `[lng, lat]` directly. Shared coordinate/station/haversine math lives in `src/lib/services/mapUtils.ts` and `gpsStation.ts`, both reading `CONST.*` from `paverate.yaml` (no hardcoded earth-radius/feet-per-meter magic numbers).
- Minimum touch target 48px for all map controls and interactive elements.
- Maps are browser-only: `MapView` lazily `import()`s `maplibre-gl` inside `onMount` behind a `browser` guard; wrap `<MapView>` usage in `{#if browser}` when a parent renders during SSR. Never CDN script injection or `window`-global map libs.
- Always check `browser` from `$app/environment` before initializing maps (SSR safety).
- GDOT route geometry is obtainable from GDOT GPAS ArcGIS (Layer 5 carries route polylines): `fetchArcgisFeatures` (`src/lib/server/dot/arcgis-fetch.ts`) already supports `returnGeometry`, and `lookupGdotBoundaries` does county/district lookup from coords. PDF imports currently DON'T use this — `from-import` hard-codes lat/lng to null and creates zero `road_sections`, so importing a PDF yields a project with no map/route until hand-drawn.

# AGENTS.md

## Learned User Preferences

- Never add `Co-authored-by: Cursor` (or any Cursor) trailers to commit messages; the user wants commit messages fully clean.
- Do not provide a list of changes in responses.
- Never create fake/sample data or data generators; treat all data as real or captured by real systems.
- Never ask the user to restart development servers.
- For UI work, keep coloring clean, simple, and non-invasive; lean on the brand slate + single yellow accent rather than heavy/decorative color.
- Keep the codebase unified: do not leave unused, duplicated, or over-complex code from multi-agent history; consolidate and remove dead code rather than piling on new files.
- Prefer reframing/inventorying what already exists over adding new features; when asked to "reimagine," audit current capabilities and decide keep/reframe/change rather than building more.
- Keep PaveRate branding visible even when an org sets a custom logo (orgs can customize name/logo/default values, but PaveRate identity stays present).
- Don't state deployment/infrastructure facts (e.g. which branch maps to the `dev.paverate.com` subdomain, what a deploy is serving) as fact from assumption or CLI metadata — verify directly (probe the live endpoint/response) before asserting; the user pushes back hard on guessed claims presented as truth. Likewise, don't declare a pipeline/import run a "success" when the review screen still surfaces warnings — the user scrutinizes overstated success, so audit each warning honestly (which are real bugs vs benign noise) before claiming it worked. Same with PDF/document data: READ the actual extracted evidence (e.g. `structurer-input/*.evidence.txt`) before theorizing about what a document does/doesn't contain — the user pushes back hard ("are you even looking at what the documents show?") when you claim a value "isn't in the PDF" without checking; the value is often present and just unparsed.
- This is a paving app, so the map is roads-only: every line/marker/section must lie on a road. No free-floating off-road points — clicks must snap to the route centerline or the real road network. The user repeatedly rejects map editors that let you drop arbitrary points anywhere.

For variable-depth mill-and-fill jobs there is NO fixed lift thickness in the document ("as directed by the engineer"); the 1½" is the MILL depth, not the asphalt lift — so DERIVE `target_thickness` from the parsed spread rate (the inverse of the existing `THICK_MULT` relationship) rather than fabricating a thickness or mistaking mill depth for it. The import "Paving Setup" pre-fill already consumes `spread_rate_lbs_sy`.

- PDF import is an LLM-PRIMARY pipeline with **AWS Bedrock (Claude Sonnet 4 via the Converse REST API) as the PRIMARY structurer and the Workers AI chain (Kimi/Llama) as automatic fallback** — Workers AI models proved inadequate (timeouts / unparseable `no-json`) on the nested `segments[]`/`events[]`/`side_roads[]` schema, while Bedrock/Sonnet 4 produced clean, complete structure. The structurer OWNS turning messy contract/plan text into ONE strict multi-section schema (`StructuredContract` in `src/lib/server/pdf/structured-contract.ts` — `route`/`county`/`midpoint`/`gross_length_mi`/`segments[]`/`events[]`/`bid_items[]`/`production_mixes[]`, each field a `ParsedField<T>` confidence envelope), then deterministic code VALIDATES (`validate-contract.ts`: measures, sane length bands, regex-vs-LLM cross-check → agree upgrades to high, disagree keeps deterministic + warns as an actionable "use AI value" using the format the import UI's `aiConflictFromWarning` parser expects) and MAPS geometry (the LLM never computes coordinates/measures). Bedrock is called fetch-only from the Worker (`src/lib/server/pdf/bedrock-structurer.ts`: `readBedrockConfig`/`bedrockSystemPrompt`/`runBedrockStructure`) — `POST https://bedrock-runtime.{region}.amazonaws.com/model/{modelId}/converse` with plain `Authorization: Bearer ${AWS_BEARER_TOKEN_BEDROCK}` (a LONG-TERM key; short-term keys require SigV4 a bare fetch can't do), NO SigV4 and NO `boto3` (Python can't run in Workers). Converse has no JSON-mode/`response_format`, so the JSON Schema is EMBEDDED in the system prompt and the model returns raw JSON text routed through the SAME `extractFlatContract`→`wrapFlatContract`→`validateContract` path (no Bedrock-specific parsing). `structureContract(ai, pages, { bedrock, models, perModelTimeoutMs })` tries Bedrock first and falls through on missing-key/error/no-json; every diagnostic + `pdf_structure_contract` log carries an `engine` field (`bedrock`|`workers-ai`). Config is centralized in `src/lib/server/pdf/llm-config.ts`: `BEDROCK_REGION`/`BEDROCK_MODEL`/`BEDROCK_MAX_TOKENS` (default 8192) are `wrangler.jsonc` vars; `AWS_BEARER_TOKEN_BEDROCK` (alias `BEDROCK_API_KEY`) is a Cloudflare PAGES SECRET set per-env (`npx wrangler pages secret put ...` for BOTH production and preview) — it takes effect only on the NEXT deploy and won't appear in generated worker types. The reference impl is `tools/build_structurer_input.py` (`--run-bedrock`, boto3 `converse`); migrating its Python `run_bedrock`/`bedrock_system_prompt` into TS was the source of truth. Two recurring structurer gotchas: (1) the model invents bid_item/mix KEY NAMES unless the schema defines field shapes — Bedrock returned `item_number`/`extension` while `wrapBidItem` read `item_id`/`bid_amount`, silently nulling all items, so keep the schema field-explicit AND make `wrapBidItem`/`wrapProductionMix` alias-tolerant (`item_number`→`item_id`, `extension`→`bid_amount`); (2) real GDOT roadway logs legitimately list MULTIPLE events at the SAME milepost (e.g. end-taper + begin-turn-lane at 2.483), so the measure check must flag only a true DECREASE (`< lastMeasure`), allowing equal — strict-increasing is wrong. The "AI assist could not supplement (superseded-by-structurer)" message is a SUCCESS no-op (gap-fill correctly skipped because the structurer applied), not a failure — keep it silent. The key data-model insight: a project is N DISCONNECTED segments (a MultiLineString), not one continuous route — `segments[]` generalizes both real document families: (a) a City/LMIG local-streets contract (e.g. `docs/Jobs/25169` Butler = 7 physically separate streets, `kind:local_street`, `measure_axis:none`, grouped by funding program LMIG/LRA, OSRM-mapped) and (b) a GDOT state-route roadway-log (e.g. `docs/Jobs/25186` SR 7 ALT = ONE route but TWO milepost-RESET sections — mainline + ramp — each its own `project_mile` axis, LRS-mapped). Structurer rules: a milepost RESET starts a NEW segment (never merge axes); `(CONTINUED)` headers are page breaks of the SAME section (strip, don't split); explode multi-road lines into multiple `side_roads[]`; ignore OCR-garbled typical-section pages. Each segment carries `name`/`kind`/`group`/`treatment`/`begin_terminus`/`end_terminus`/`length_mi`/`geometry_confidence`; un-geocodable termini ("end of pavement", "Beginning of Curve") get a best-effort snapped line flagged low-confidence — never a fabricated point. DIRECTION the user wants this pipeline to move (format-agnostic, AI-PRIMARY): the current design is too format-dependent (`detectDocumentType()` sentinel regexes + a `projectImportCandidate` gate that only lets the AI run on KNOWN doc types, verbiage-exact regex as the primary extractor, AI as secondary fill-only, text-only with rendered diagram images never sent, the prompt told to "Ignore OCR-garbled typical-section pages", and the `StructuredContract` schema lacks any pavement-structure/spread-rate/per-section-mile-range field — so typical-section data is structurally uncapturable, and there is no real OCR for scanned PDFs). The intended architecture inverts this: LLM-primary, format-agnostic extraction over the full per-page package (text + diagram images for low-text/diagram pages, hybrid), returning required fields with confidence + source-page refs, with the regex parser demoted to a cross-check/validator (NOT a gate). Bedrock Claude Sonnet 4 via Converse supports image content blocks (`{ image: { format, source: { bytes } } }`) alongside the existing `{ text }` block, so adding vision is a natural extension of the current `runBedrockStructure` call.
- Multi-segment job persistence reuses the EXISTING `road_sections` table (it is already a per-segment, geometry-bearing, job-site-scoped entity — NO new table needed); migration `0078_road_section_segments.sql` added nullable `segment_group`/`treatment`/`measure_axis`/`begin_terminus`/`end_terminus`/`geometry_confidence` + a `(job_site_id, segment_group)` index. `from-import/+server.ts` takes an optional `segments[]`: when present it writes ONE `road_sections` row per disconnected segment (own geometry/group/treatment, independent station-0 axis) and pins the job at the first segment's centroid, skipping `upsertJobSiteRoute` (the segments ARE the geometry); when absent the existing single-route slicing path is unchanged. The pure segment→row mapping lives in `src/lib/server/import-segments.ts` (`buildSegmentRows`, unit-testable without D1/AI); the INSERT uses the core/optional column-split + retry-without-optional guard for the deploy-ordering trap.

## Map Conventions

- The map layer is fully on MapLibre GL JS: the ONLY map layer is `src/lib/components/map-v2/` (lazy, SSR-safe `MapView` + `MapMarker`/`MapPolyline`/`MapPolygon`/`MapGeoJSON`/`MapPopup`/`RoadProgressLayer`). The old `src/lib/components/map/` Leaflet directory has been removed, and `leaflet`/`@types/leaflet` are no longer dependencies (do NOT re-add them or `ssr.noExternal: ['leaflet']`). Build new map features by composing `map-v2` components inside a `<MapView>`; never use raw `L.map()` or any Leaflet API.
- Store geometry as GeoJSON TEXT in D1 (column name: `geometry_geojson`); use `geoJsonToD1()` / `d1ToGeoJson()` from `src/lib/services/mapUtils.ts`.
- Coordinates: `map-v2` component props take `[lat, lng]` (the component flips internally); geometry is stored as GeoJSON `[lng, lat]` per RFC 7946, and `RoadProgressLayer` consumes stored `[lng, lat]` directly. Shared coordinate/station/haversine math lives in `src/lib/services/mapUtils.ts` and `gpsStation.ts`, both reading `CONST.*` from `paverate.yaml` (no hardcoded earth-radius/feet-per-meter magic numbers).
- Minimum touch target 48px for all map controls and interactive elements.
- Maps are browser-only: `MapView` lazily `import()`s `maplibre-gl` inside `onMount` behind a `browser` guard; wrap `<MapView>` usage in `{#if browser}` when a parent renders during SSR. Never CDN script injection or `window`-global map libs.
- Always check `browser` from `$app/environment` before initializing maps (SSR safety).
- GDOT route geometry is obtainable from GDOT GPAS ArcGIS (Layer 5 carries route polylines): `fetchArcgisFeatures` (`src/lib/server/dot/arcgis-fetch.ts`) supports `returnGeometry`, and `lookupGdotBoundaries` does county/district lookup from coords. PDF import NOW uses this — `from-import` resolves real coordinates (GDOT route polyline centroid → US Census geocode → GDOT county-centroid fallback, in `src/lib/server/gdot-geometry.ts`), persists `gdot_county`/`gdot_district`, stores the route as waypoints, and auto-creates stationed `road_sections` from the route geometry. The GPAS `COUNTY` field is often a numeric code, so a `COUNTY LIKE '%NAME%'` filter can wrongly exclude every row — retry the route query WITHOUT the county filter. Abandoned: reprojecting the contract PDF's `MID-POINT COORDINATES ... WEST ZONE` State Plane value with a hand-rolled transform (landed in the wrong state on every parameter guess) — use a real tested `proj4` + verified GDOT EPSG if that pin source is ever needed, never a guessed transform.
- Roads-only is enforced in the map editors (this is a paving app — every line/marker/section must lie on a road): `TerminusPicker`/`MapStationPicker` snap clicks to the route centerline via `coordinateToStation`/`stationToCoordinate` (re-snapping the marker back onto the road); `RoadSectionEditor` builds sections by slicing the route polyline between two snapped stations (no route ⇒ can't add sections); `RouteAlignmentMap` treats taps as control points snapped to the real road network and routes along roads via `src/lib/services/roadSnap.ts` (key-less public OSRM `nearest`+`route`, returns null on failure, never fabricates geometry — surface "tap on a road" rejection in the UI). The review page distinguishes a filled-but-low field (amber "verify" with a one-click Confirm that does NOT change the value) from an empty low field (red "needs input"). `RouteAlignmentMap` accepts a read-only `segments` prop (the import pipeline's `mapped_segments` — real per-segment GeoJSON centerlines resolved from GDOT LRS/ArcGIS) and overlays them as `MapGeoJSON` lines colored by `geometry_confidence` (high=green / medium=blue / low=orange); the import server computes `route_preview.mapped_segments` but the import UI historically dropped them at the type boundary — surface real geometry on the import review map, don't discard it. Geometry math is consolidated on scoped `@turf/*` packages (buffer/length/centroid) while keeping station math config-driven (`CONST.*`) and snapping roads-only; deferred `route-snapper` (WASM + per-area graph cost) and Terra Draw (conflicts with roads-only). OpenFreeMap tiles have a Workbox runtime-cache rule so the map works offline.

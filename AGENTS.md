# AGENTS.md

## Learned User Preferences

- Never add `Co-authored-by: Cursor` (or any Cursor) trailers to commit messages; the user wants commit messages fully clean.
- Do not provide a list of changes in responses.
- Never create fake/sample data or data generators; treat all data as real or captured by real systems.
- Never ask the user to restart development servers.
- For UI work, keep coloring clean, simple, and non-invasive; lean on the brand slate + single yellow accent rather than heavy/decorative color.
- Keep the codebase unified: do not leave unused, duplicated, or over-complex code from multi-agent history; consolidate and remove dead code rather than piling on new files.
- Prefer reframing/inventorying what already exists over adding new features; when asked to "reimagine," audit current capabilities and decide keep/reframe/change rather than building more.
- Keep PaveRate branding visible even when an org sets a custom logo (orgs can customize name/logo/default values, but PaveRate identity stays present).
- Don't state deployment/infrastructure facts (e.g. which branch maps to the `dev.paverate.com` subdomain, what a deploy is serving) as fact from assumption or CLI metadata — verify directly (probe the live endpoint/response) before asserting; the user pushes back hard on guessed claims presented as truth.
- This is a paving app, so the map is roads-only: every line/marker/section must lie on a road. No free-floating off-road points — clicks must snap to the route centerline or the real road network. The user repeatedly rejects map editors that let you drop arbitrary points anywhere.

## Learned Workspace Facts

- The shell is PowerShell on Windows: bash idioms like `head`, `ls -la`, `du`, and inline `$VAR=...` env-prefixes fail. Use `Select-Object`, `Get-ChildItem`, and run multi-statement logic from a script file (passing complex `$`-laden commands inline often gets the `$` stripped).
- Build script must stay `wrangler types && vite build` (no `--check`); `--check` breaks the Cloudflare Pages build because `worker-configuration.d.ts` does not exist pre-build.
- CI (`.github/workflows/test.yml`) runs separate jobs: Unit, Integration, Component (vitest, projects under `vitest.workspace`), E2E (Playwright), and Lint+Type Check (`svelte-check`). Known recurring breakers: (1) `npm ci` needs Node ≥ 22 — a transitive dep (`@cloudflare/kv-asset-handler`) requires it, so the workflow must use Node 22 (local passes on 22+); (2) coverage thresholds were aspirational 80/60/50% that were NEVER met and failed every commit — they're now realistic per-project FLOORS (ratchet up over time, don't fabricate tests or game them); (3) `package-lock.json` must stay in sync (`npm ci` fails on drift, e.g. missing `@vitest/coverage-v8`, or after a deps add like the `@turf/*` cluster); (4) the Playwright E2E `webServer` (runs `vite dev`) times out at 60s in CI — pre-existing infra issue (dev server slow/failing to bind under adapter-cloudflare platform proxy), not a code bug. As of the tech-debt cleanup, `svelte-check` is now FULLY CLEAN (0 errors / 0 warnings) — the ~85 prior type errors (mostly untyped `res.json()` → `'data' is of type 'unknown'` in `*.integration.test.ts`/routes) and ~136 a11y warnings were all fixed, so KEEP it clean (don't reintroduce untyped `res.json()`; for intentional "snapshot props once into editable form state" cases use a documented `svelte-ignore` rather than wrongly converting to `$derived`). The real Cloudflare `vite build` and in-IDE ReadLints remain authoritative for changed files; `maplibre-gl` must be installed in `node_modules` (it's in `package.json`) or the build fails to resolve it.
- All values, spec rates, constants, theme tokens, and labels live in the single config `src/lib/config/paverate.yaml`; formulas live in `src/lib/config/formulas.ts`. No magic numbers in code — every value is read by its Validation Matrix ID (e.g. `CONST.LB_PER_TON`, `TACK.NEW_AC`).
- Calculations must be traceable to GDOT specs via `docs/validation-matrix.md` and `docs/VALIDATION.md`; verified spec excerpts are saved under `docs/spec-excerpts/`.
- GDOT spec PDFs live in `docs/` (2021 Standard Specs, 2024 Supplemental, `cm001` Construction Manual); `tools/spec_search.py` (uses `pypdf`, in `requirements.txt`) searches/dumps them. The PDF page index is offset from the printed footer page number (e.g. printed p.449 = PDF p.459).
- Brand identity: dark slate `#2e3b46` + single yellow accent `#f2c037`; dark-mode default, high contrast for outdoor use, mobile-first with 48px+ touch targets. Brand/PWA assets are generated by `tools/make_brand_assets.py` (Pillow/`PIL`) from source transparent PNGs in `branding/`, emitting trimmed icons/logos to `static/icons/` and `static/`; in-app logos are inline brand-colored SVG (slate badge + yellow road/lane) to keep transparent corners sharp.
- Stack: SvelteKit 5 (runes) + `@sveltejs/adapter-cloudflare`, PWA via `@vite-pwa/sveltekit`, YAML imported at build time via `@rollup/plugin-yaml`; deploys to Cloudflare Pages from `main`. Accounts/orgs/jobsites are a planned optional cloud layer (Workers + D1 + R2), never gating offline calculator use. Runes gotcha: never name a component prop `state` — it shadows the `$state` rune so `$state(...)` is misread as a store subscription to the prop; rename the prop (e.g. `data`).
- Auth/accounts use a Cloudflare D1 binding (`event.platform.env.DB`) and an R2 binding `ASSETS_BUCKET` (bucket `paverate-org-assets`, used for org logo uploads); both are available under `vite dev` via adapter-cloudflare's platform proxy reading `wrangler.jsonc`. A dev-only login endpoint/button seeds a known dev user+org and is gated behind SvelteKit's `dev` flag (absent in production). `setSessionCookie` (`src/lib/server/auth.ts`) sets `secure: !dev` so local HTTP sessions persist — a hard-coded `Secure` cookie is silently dropped over `http://localhost` and 401s every protected route.
- Local D1 is bootstrapped via npm scripts (no manual per-file applies): `npm run db:local` (apply all `migrations/0*.sql` in sorted order via `scripts/db-local.ps1`), `db:local:reset` (wipe + rebuild), and `db:pull-dev` (export the remote dev D1 to `.wrangler/dev-snapshot.sql` and load it into local, for real-data testing). No D1 mock/refactor — `DbHelper` issues real SQL via `prepare().bind().first/all/run()` and adapter-cloudflare gives `vite dev` a real local SQLite.
- Protected `+server.ts` endpoints must `return error` (not `throw error`) when catching a `Response` thrown by `requireAuth`/`requireOrgRole`/`requireGlobalAdmin`; SvelteKit only special-cases thrown `Redirect`/`HttpError`, so a thrown raw `Response` becomes a fatal 500 instead of the intended 401/403 JSON.
- The product has pivoted from a simple calculator into a full field-paving project-management tool; "Job Sites" are really Projects (config, costs/contract value, crews, equipment, route/geometry, daily logs, loads), with calculators as one piece and most values auto-derived. Keep a standalone one-off calculator too. The DB/APIs already support the PM model; the UI lags it. Orgs can customize name/logo/default values, with logo uploads via the `ASSETS_BUCKET` R2 binding noted above. The `/app` route is "Quick Calc" — it should be pure calculators/inputs only (no daily-log/"Today"/project-management features, which belong to the separate Job Sites/dashboard routes).
- Shared calculator inputs flow through `src/lib/stores/calcContext.svelte.ts`, the single source of truth that merges manual overrides → job-site config → the `job` store (`src/lib/stores/job.svelte.ts`) with source tracking. A value set once (e.g. `lift_thickness`, `road_width`, `course_type`) should be read by EVERY calculator via `calcContext`, not directly from `job` — several cards historically read `job.*` directly and so ignored overrides, which is the bug to standardize away. Genuinely job-only fields (machine, firstPass, waste, truck load) still read from `job`.
- Server-side PDF text extraction uses `pdfjs-serverless` (`getDocument` → `getTextContent`, with `disableFontFace`/`isEvalSupported:false`/`useSystemFonts`), NOT `unpdf`: `unpdf`'s bundled PDF.js works in `wrangler dev` locally but throws `Object.defineProperty called on non-object` ("Serverless PDF.js bundle could not be resolved") in the actual deployed Cloudflare Pages runtime, so it was replaced. Uploaded source PDFs are stored in the `ASSETS_BUCKET` R2 binding (e.g. under `imports/<orgId>/<uuid>.pdf`). The GDOT job-setup / contract-summary parser (`src/lib/server/pdf/parse-gdot.ts`) is 100% deterministic regex/zone parsing (no LLM, leaves unknowns undefined, never invents data): `detectDocumentType()` classifies via sentinel-phrase regex, parsed values are wrapped in a `ParsedField<T>` confidence envelope (high/medium/low) with low-confidence regions collected as `lowConfidenceZones`, and the no-text path returns 422 ("may be a scanned image"). The Phase-2 Workers AI LLM fallback is now BUILT (`src/lib/server/pdf/llm-fallback.ts`, JSON-Mode): it runs server-side during import, gated by `needsLlmFallback()` (a geographic/identity field is low/null) AND the `AI` binding being present AND a usable JSON response; it fills ONLY low/null fields via `mergeField` (never overrides medium/high), stamps `source:'llm-fallback'`, and degrades silently. `no-zone-text`/`ai-binding-unavailable`/`no-low-confidence-fields` are benign NO-OPs (outcome `not-needed`/`binding-unavailable`), NOT failures — only a real model error/no-JSON is a `failed` outcome that warns. Workers AI bindings are typically unavailable under local `vite dev` (so it reports "not needed/unavailable" locally); the `AI` binding lives in `wrangler.jsonc`. Prefer `-fast`/`fp8-fast` model variants — `@cf/meta/llama-3.1-8b-instruct` is deprecated (May 2026) and Cloudflare's documented JSON-Mode supported-models list is contested. Rendering PDF pages to images (schematics) can NOT run in the Workers runtime (no canvas) — do it client-side with `pdfjs` in the browser and upload the PNGs to R2.
- Migrations live in `migrations/` (NOT the nested `migrations/migrations/`) and are applied by `scripts/db-local.ps1`, which now runs `npx wrangler d1 migrations apply paverate-db --local` (tracked in `d1_migrations`); `-Reset` drops all user objects by name first (the SQLite file is locked while `vite dev` runs). This requires strict sequential numbering with NO duplicates — verify the current head before adding a new file (recent head was `0078`). SQLite has no `ADD COLUMN IF NOT EXISTS`, so keep a bare `ALTER TABLE ... ADD COLUMN` in its own migration file. Deploying code that writes a new column BEFORE its migration is applied to the shared remote D1 causes a live 500 (e.g. `from-import` "no such column"); guard by building SQL dynamically with a core/optional column split + retry-without-optional. `upsertJobSiteConfig` does NOT persist the cost fields (`total_tonnage`/`cost_per_ton`/`total_contract_value`) — a pre-existing gap. (Historical multi-agent dupes like `0024`/`0025`/`0046`/`0071`/`0074` predate the numbering rule and should be cleaned up rather than added to.)
- Wrangler is currently authenticated (as `dylan@dlewis.me`, with `d1`/`pages` write scope), so live Cloudflare access (remote D1 migrations, Pages deployment status via the API) works from the shell — the older "expired OAuth token" note is stale. There is exactly ONE D1 database in the account (`paverate-db`, single `DB` binding in `wrangler.jsonc`, no per-environment D1), so `dev.paverate.com` (a Pages preview/branch deploy) and production SHARE the same database — there is no separate "dev D1"; any migration lag is on that one shared DB. Migrations auto-apply only on push to `main` via `.github/workflows/deploy.yml` (now using the tracked `npx wrangler d1 migrations apply paverate-db --remote`, which only runs untracked files and fails loudly — the old "execute every .sql file, swallow errors" loop hid broken migrations); `npm run db:migrate:remote` is the manual equivalent. Check pending with `wrangler d1 migrations list paverate-db --remote`. Watch for 500s on org endpoints (`/api/org/crew-status`, `/api/org/settings`) sharing `getOrgByUserId` → `getUserRole` → `getOrgSettings`. To verify whether a Cloudflare PAGES build actually passed/failed, do NOT rely on `wrangler pages deployment list` (it lists deployments and URLs but does NOT clearly flag build pass/fail) — query the CF REST API for the deployment's `latest_stage`/per-stage status (build/deploy `success` vs `failure`), authenticating with the OAuth token from the local wrangler config since the account is OAuth- (not API-key-) authed. Cloudflare Pages build failures share the SAME root causes as the GitHub pipeline (Node engine, out-of-sync `package-lock.json`); a green local `npm run build` plus a synced lockfile usually means Pages is green too.
- PWA `navigateFallback` must stay `null` in `vite.config.ts`: this is an SSR app on Cloudflare with nothing prerendered at `/`, so the default `navigateFallback: '/'` makes Workbox bind a `NavigationRoute(createHandlerBoundToURL('/'))` to a non-precached URL and throw `non-precached-url :: [{"url":"/"}]` on every navigation (non-fatal but noisy). Asset precaching + font/`/api` runtime caching are unaffected. Separately, Leaflet must NOT be listed in `build.rollupOptions.external` in `vite.config.ts` — externalizing it emits a bare `import "leaflet"` the browser can't resolve ("Failed to resolve module specifier 'leaflet'") in the production/preview build (dev hides it because Vite resolves bare specifiers on the fly); keep `ssr.noExternal: ['leaflet']` so it's bundled for both client and SSR.
- GDOT-import projects are multi-scope/multi-mix: each mix carries TWO distinct quantities — `bid_quantity` = "allotted" (contract/Paving-Sections quantity the state/customer pays for) vs `takeoff_tonnage` = "target" (the company's Production-Goals number we aim to place) — plus a `contract_unit_price` (from the matching asphalt bid item) so margin = unit price − our cost/ton. Don't color the target-vs-allotted variance good/bad (placing fewer tons than allotted is just less line revenue, not "more money"; margin comes from cost < unit price). The parser must map mix names to the mix_type enum (not store the name as the type) and detect/report which document was uploaded (Contract Summary vs Job Setup), warning when one is missing.
- App shell layout (`src/lib/components/shell/AppShell.svelte` + `NavSidebar.svelte`) is a responsive CSS grid with breakpoints at 900px (icon-rail sidebar) and 1100px (labeled sidebar); mobile uses a hamburger drawer. The grid MUST use explicit `grid-template-areas` (`nav` spanning all rows, with `weather`/`main`/`footer`/`context` stacked in the content column) — relying on auto-flow scatters main/footer into the sidebar column. The `.sidebar` must be `position: sticky; align-self: start` across the entire ≥900px range (not just 900–1099px), or at ≥1100px it falls back to base `position: fixed`, drops out of grid flow, and leaves an empty reserved column.
- PDF import is moving to an LLM-PRIMARY pipeline: a stronger Workers AI model OWNS structuring the messy contract/plan text into ONE strict multi-section schema (`StructuredContract` in `src/lib/server/pdf/structured-contract.ts` — `route`/`county`/`midpoint`/`gross_length_mi`/`segments[]`/`events[]`/`bid_items[]`/`production_mixes[]`, each field a `ParsedField<T>` confidence envelope), then deterministic code VALIDATES (`validate-contract.ts`: monotonic measures, sane length bands, regex-vs-LLM cross-check → agree upgrades to high, disagree keeps deterministic + warns) and MAPS geometry (the LLM never computes coordinates/measures). The model is centralized in `src/lib/server/pdf/llm-config.ts` (no longer hardcoded across files; verify the current strongest JSON-Mode Workers AI instruct model against live Cloudflare docs). The key data-model insight: a project is N DISCONNECTED segments (a MultiLineString), not one continuous route — `segments[]` generalizes both real document families: (a) a City/LMIG local-streets contract (e.g. `docs/Jobs/25169` Butler = 7 physically separate streets, `kind:local_street`, `measure_axis:none`, grouped by funding program LMIG/LRA, OSRM-mapped) and (b) a GDOT state-route roadway-log (e.g. `docs/Jobs/25186` SR 7 ALT = ONE route but TWO milepost-RESET sections — mainline + ramp — each its own `project_mile` axis, LRS-mapped). Structurer rules: a milepost RESET starts a NEW segment (never merge axes); `(CONTINUED)` headers are page breaks of the SAME section (strip, don't split); explode multi-road lines into multiple `side_roads[]`; ignore OCR-garbled typical-section pages. Each segment carries `name`/`kind`/`group`/`treatment`/`begin_terminus`/`end_terminus`/`length_mi`/`geometry_confidence`; un-geocodable termini ("end of pavement", "Beginning of Curve") get a best-effort snapped line flagged low-confidence — never a fabricated point.
- Multi-segment job persistence reuses the EXISTING `road_sections` table (it is already a per-segment, geometry-bearing, job-site-scoped entity — NO new table needed); migration `0078_road_section_segments.sql` added nullable `segment_group`/`treatment`/`measure_axis`/`begin_terminus`/`end_terminus`/`geometry_confidence` + a `(job_site_id, segment_group)` index. `from-import/+server.ts` takes an optional `segments[]`: when present it writes ONE `road_sections` row per disconnected segment (own geometry/group/treatment, independent station-0 axis) and pins the job at the first segment's centroid, skipping `upsertJobSiteRoute` (the segments ARE the geometry); when absent the existing single-route slicing path is unchanged. The pure segment→row mapping lives in `src/lib/server/import-segments.ts` (`buildSegmentRows`, unit-testable without D1/AI); the INSERT uses the core/optional column-split + retry-without-optional guard for the deploy-ordering trap.

## Map Conventions

- The map layer is fully on MapLibre GL JS: the ONLY map layer is `src/lib/components/map-v2/` (lazy, SSR-safe `MapView` + `MapMarker`/`MapPolyline`/`MapPolygon`/`MapGeoJSON`/`MapPopup`/`RoadProgressLayer`). The old `src/lib/components/map/` Leaflet directory has been removed, and `leaflet`/`@types/leaflet` are no longer dependencies (do NOT re-add them or `ssr.noExternal: ['leaflet']`). Build new map features by composing `map-v2` components inside a `<MapView>`; never use raw `L.map()` or any Leaflet API.
- Store geometry as GeoJSON TEXT in D1 (column name: `geometry_geojson`); use `geoJsonToD1()` / `d1ToGeoJson()` from `src/lib/services/mapUtils.ts`.
- Coordinates: `map-v2` component props take `[lat, lng]` (the component flips internally); geometry is stored as GeoJSON `[lng, lat]` per RFC 7946, and `RoadProgressLayer` consumes stored `[lng, lat]` directly. Shared coordinate/station/haversine math lives in `src/lib/services/mapUtils.ts` and `gpsStation.ts`, both reading `CONST.*` from `paverate.yaml` (no hardcoded earth-radius/feet-per-meter magic numbers).
- Minimum touch target 48px for all map controls and interactive elements.
- Maps are browser-only: `MapView` lazily `import()`s `maplibre-gl` inside `onMount` behind a `browser` guard; wrap `<MapView>` usage in `{#if browser}` when a parent renders during SSR. Never CDN script injection or `window`-global map libs.
- Always check `browser` from `$app/environment` before initializing maps (SSR safety).
- GDOT route geometry is obtainable from GDOT GPAS ArcGIS (Layer 5 carries route polylines): `fetchArcgisFeatures` (`src/lib/server/dot/arcgis-fetch.ts`) supports `returnGeometry`, and `lookupGdotBoundaries` does county/district lookup from coords. PDF import NOW uses this — `from-import` resolves real coordinates (GDOT route polyline centroid → US Census geocode → GDOT county-centroid fallback, in `src/lib/server/gdot-geometry.ts`), persists `gdot_county`/`gdot_district`, stores the route as waypoints, and auto-creates stationed `road_sections` from the route geometry. The GPAS `COUNTY` field is often a numeric code, so a `COUNTY LIKE '%NAME%'` filter can wrongly exclude every row — retry the route query WITHOUT the county filter. Abandoned: reprojecting the contract PDF's `MID-POINT COORDINATES ... WEST ZONE` State Plane value with a hand-rolled transform (landed in the wrong state on every parameter guess) — use a real tested `proj4` + verified GDOT EPSG if that pin source is ever needed, never a guessed transform.
- Roads-only is enforced in the map editors (this is a paving app — every line/marker/section must lie on a road): `TerminusPicker`/`MapStationPicker` snap clicks to the route centerline via `coordinateToStation`/`stationToCoordinate` (re-snapping the marker back onto the road); `RoadSectionEditor` builds sections by slicing the route polyline between two snapped stations (no route ⇒ can't add sections); `RouteAlignmentMap` treats taps as control points snapped to the real road network and routes along roads via `src/lib/services/roadSnap.ts` (key-less public OSRM `nearest`+`route`, returns null on failure, never fabricates geometry — surface "tap on a road" rejection in the UI). The review page distinguishes a filled-but-low field (amber "verify" with a one-click Confirm that does NOT change the value) from an empty low field (red "needs input"). Geometry math is consolidated on scoped `@turf/*` packages (buffer/length/centroid) while keeping station math config-driven (`CONST.*`) and snapping roads-only; deferred `route-snapper` (WASM + per-area graph cost) and Terra Draw (conflicts with roads-only). OpenFreeMap tiles have a Workbox runtime-cache rule so the map works offline.
